---

## 0. One-line definition

What is this workflow / discipline / methodology in a single sentence?

## 1. Why this exists

The problem it solves, the cost of not having it, the trigger that prompted writing it down.

## 2. When it applies

Specific situations, project types, or signals that should trigger using this workflow.
Equally important: when it does NOT apply.

## 3. Core rules

The actual content. Bullet list of rules, decisions, or steps. Be concrete.

- Rule 1: ...
- Rule 2: ...

## 4. Examples

Concrete examples grounded in real situations. Avoid abstract toy cases.

## 5. Anti-patterns

What this rule explicitly rejects, with brief reasoning. The "why not" is often more
useful than the "why".

## 6. Integration with other workflows

How this interacts with other entries in `workflow/`, `queen/`, or `projects/`.
Cross-reference relevant files.

## 7. Maintenance

- Last reviewed: YYYY-MM-DD
- Known stale conditions: what would invalidate this entry
- Promotion path: is any part of this exportable to public surfaces (with desensitization)?

---

## Notes for the author

- Lead with conclusions, not narrative.
- No filler. Trim mercilessly.
- If you cannot describe it in ≤200 lines, it is two topics. Split.
- If it depends on a specific employer/client/codename, it does not belong in `workflow/` —
  it belongs in `projects/` or stays in agent memory.
